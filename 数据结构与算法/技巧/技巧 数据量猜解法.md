**Big O 时间复杂度一样，实际运行时间通常也差不多，但常数可能不同**

维护数组最大值和最小值：

```
int mn = INF, mx = -INF;
for (int x : a) {
    mn = min(mn, x);
    mx = max(mx, x);
}
```

这是一次遍历，时间复杂度：

```
O(n)
```

分两遍：

```
int mn = INF;
for (int x : a) mn = min(mn, x);

int mx = -INF;
for (int x : a) mx = max(mx, x);
```

也是：

```
O(n) + O(n) = O(2n) = O(n)
```

所以从**渐进复杂度**看，它们一样，都是 `O(n)`。

但从**实际常数**看，一次遍历通常更好一点，因为：

- 只扫一遍数组，内存访问更少；
- cache locality（缓存局部性）更好；
- 循环控制开销少一半；
- 对大数组更明显。

不过差距通常不是复杂度层面的，而是常数层面的。小数据时几乎没区别。

推广一下，主人说的这个理解基本正确：

```
一个 O(1) 的大常数操作，拆成有限个 O(1) 小操作，整体仍然是 O(1)。
一个 O(n) 操作做有限次，整体仍然是 O(n)。
```

形式上：

```
O(cn) = O(n)    其中 c 是固定常数
O(c) = O(1)     其中 c 是固定常数
```

关键限制是：**“有限个”必须是和输入规模 n 无关的常数**喵。

比如：

```
扫 2 遍数组：O(2n) = O(n)
扫 10 遍数组：O(10n) = O(n)
扫 1000000 遍数组：理论仍是 O(n)，但实际可能很慢
扫 n 遍数组：O(n^2)，这就变了
```

所以竞赛里判断复杂度时要分两层：

- **算法复杂度层面**：一次维护 max/min 和两次分别维护，都是 `O(n)`。
- **工程性能层面**：一次遍历通常更优，尤其数组很大时。

复杂度分析：时间 `O(n)`，空间 `O(1)`。
